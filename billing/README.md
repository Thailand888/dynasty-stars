# Учёт расхода токенов Hermes → счёт клиенту

Скрипт `token_report.py` читает базу сессий Hermes Agent
(`~/.hermes/state.db`, таблица `sessions`) **только на чтение** (`mode=ro`,
запись исключена) и считает расход токенов и его стоимость в USD и RUB.

## Запуск

```bash
cd /Users/thai/poodle/billing

# Весь период
python3 token_report.py

# За период (включительно по обеим границам)
python3 token_report.py --from 2026-06-07 --to 2026-06-11

# Свои цены / курс
python3 token_report.py --in-price 15 --out-price 75 --rub-rate 92

# Другая база
python3 token_report.py --db /path/to/state.db
```

`python3` — системный (`/usr/bin/python3`), `sqlite3` входит в stdlib,
внешних зависимостей нет.

## Параметры

| Флаг | Значение по умолчанию | Описание |
|---|---|---|
| `--from` | начало данных | дата начала `YYYY-MM-DD` (включительно) |
| `--to` | конец данных | дата конца `YYYY-MM-DD` (включительно, до 23:59:59) |
| `--in-price` | `15.0` | USD за 1M входных токенов (OpenRouter, Claude Opus) |
| `--out-price` | `75.0` | USD за 1M выходных токенов |
| `--rub-rate` | `90.0` | курс RUB за 1 USD |
| `--db` | `~/.hermes/state.db` | путь к базе |

## Откуда берутся данные

Таблица `sessions`, поля: `input_tokens`, `output_tokens`,
`cache_read_tokens`, `cache_write_tokens`, `reasoning_tokens`,
`started_at` (Unix-время), `source` (платформа: cli/telegram/cron/tui),
`model`, `estimated_cost_usd`/`actual_cost_usd`.

Тарифицируются **входные + выходные** токены по заданным ценам.
Cache / reasoning токены показаны **справочно**.

## ⚠️ Важно про стоимость и кэш

В отчёте есть строка «оценка провайдера». Она обычно **выше** нашего
расчёта по input/output, потому что Claude Opus тарифицирует ещё и
**кэш** (cache read ~$1.5/1M, cache write ~$18.75/1M), а в базе накоплены
сотни миллионов кэш-токенов. Варианты для счёта клиенту:

1. **Простой счёт** (по умолчанию скрипта) — только input/output по
   $15/$75. Прозрачно и понятно клиенту.
2. **Точный счёт** — взять «оценку провайдера» (`actual/estimated_cost_usd`),
   она уже включает кэш и реальные коэффициенты OpenRouter. Это самая
   честная сумма по факту биллинга.

Рекомендация: для счёта владельцу питомника используйте «оценку
провайдера» как базу, а разбивку по input/output — как пояснение.

## 🔑 РЕКОМЕНДАЦИЯ: как тегировать работу по проектам/клиентам

Сейчас Hermes **не размечает** сессии по проектам, поэтому отчёт даёт
ОБЩИЙ расход + разбивку по дням и платформам. Чтобы выставлять счёт по
конкретному клиенту, нужно научиться отделять «его» сессии. Варианты по
возрастанию надёжности:

1. **Отдельный профиль Hermes на клиента/проект (лучший вариант).**
   Запускать работу по клиенту в своём профиле
   (`~/.hermes/profiles/<client>/`) — у каждого профиля своя `state.db`.
   Тогда счёт = отчёт по базе конкретного профиля:
   `python3 token_report.py --db ~/.hermes/profiles/poodle/state.db`.
   Полная изоляция, ничего не перемешивается.

2. **Отдельный рабочий каталог (cwd) на проект.** Поле `sessions.cwd`
   уже пишется. Если работу по клиенту вести из выделенной папки
   (напр. `/Users/thai/poodle/`), отчёт можно фильтровать по `cwd LIKE`.
   Добавить флаг `--cwd /Users/thai/poodle` в скрипт — несложно.

3. **Разбивка по временным окнам.** Если работа по клиенту шла в
   известные дни — использовать `--from/--to`. Грубо, но работает уже
   сейчас (см. разбивку по дням в отчёте).

4. **Тег в названии сессии.** Поле `sessions.title` уникально; можно
   договориться о префиксе вида `poodle: ...` и фильтровать по нему.

Для биллинга питомника рекомендую вариант **1 (профиль на клиента)** —
он не требует доработки скрипта и даёт юридически чистое разделение.
